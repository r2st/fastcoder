💼 Business Requirements

Market-driven requirements derived from DORA AI Tensions report, METR SWE-bench maintainer study, and competitive analysis of GitHub Copilot, Lovable, Claude Code, and Cursor.

BR-1: Value Proposition

Core Thesis The product is a configurable autonomy + evidence system — not a better model. It spans developer productivity, testing automation, and DevOps/ALM governance as a convergence product.
  • BR-1.1: Accept a natural-language user story and produce a working, tested, deployed, merge-ready pull request with full evidence bundle — without human intervention for low-risk changes.
  • BR-1.2: Operate on existing production codebases (multi-year, multi-repo), not just greenfield projects. Lovable's inability to import existing repos is the competitive gap this fills.
  • BR-1.3: Reduce the "verification tax" identified by DORA — time saved in code creation currently re-allocated to auditing rather than shipping net-new value.
  • BR-1.4: Achieve a PR merge rate ≥75% by maintainers, exceeding the 50% baseline from the METR SWE-bench study where AI PRs that passed automated grading were still rejected by maintainers.
  • BR-1.5: Enterprise buyers purchase controlled speed, not raw speed. Verification and governance are first-class concerns, not afterthoughts.

TR-1: Orchestrator State Machine

The core pipeline is a 10-state finite state machine that processes each user story through the full development lifecycle. The agent converges — it never just finishes.

Loop-First Architecture On failure at any stage, the orchestrator rolls back to the appropriate state with enriched context (error details, stack traces, previous attempts) and retries with progressive strategy escalation. Max 5 iterations with oscillation detection.
  • TR-1.1: Convergence detection with SHA-256 fingerprinting of errors to detect oscillation (same error repeating).
  • TR-1.2: Retry policy with exponential backoff, max 5 attempts, and progressive context enrichment at each attempt level.
  • TR-1.3: Adapter layer bridging 13 component interfaces to the Orchestrator's Protocol definitions.
  • TR-1.4: Story state persistence — full story state serializable for crash recovery and audit.

Evidence Bundle Generation

Every PR produced by the agent includes a structured evidence bundle:

  • Test Report: Pass/fail counts, coverage percentage, failure details with stack traces.
  • SAST Report: Security scan results with severity classifications and CWE identifiers.
  • Change Impact Analysis: Files changed, dependency graph impact, blast radius visualization.
  • Acceptance Criteria Map: Each BDD criterion linked to its verifying test(s).
  • Cost Summary: Tokens used, model calls, total cost for this story.
  • Reviewer Checklist: Pre-filled checklist of what the agent verified vs. what needs human review.

Approval Gates (7 configurable points)

  • pre_code: Review execution plan before code generation begins.
  • pre_deploy: Review changes and evidence bundle before deployment.
  • pre_production: Additional approval for production (vs staging) deployment.
  • budget_exceeded: Triggered when cost limits are hit.
  • ambiguity_detected: Triggered when story analysis finds unclear requirements.
  • blast_radius NEW: Change impact exceeds configured threshold.
  • security_finding NEW: SAST detects finding above configured severity.

Cost Budget Engine NEW

  • Per-story budget: Default $5.00, configurable. Agent pauses when exceeded.
  • Per-day budget: Default $100.00. Prevents runaway costs from batch processing.
  • Per-month budget: Default $2,000.00. Hard cap with admin notification.
  • Cost-per-token tracking: Real-time cost accumulation across all model calls per story.
  • SLA envelopes: Configurable max latency per task type with auto-fallback on timeout.

TR-6: Error Recovery & Post-Mortem Learning UPDATED

Error Taxonomy (8 categories)

  • Syntax Error, Type Error, Import/Dependency Error, Logic Error, Runtime Error, Test Failure, Build Failure, Infrastructure Error
  • SHA-256 fingerprinting for oscillation detection; progressive context enrichment over 5 retry levels.

Post-Mortem Learning Loops NEW

  1. Classify failure mode: Map to error taxonomy and determine root cause category.
  2. Update repo heuristics: If convention violation caused failure, update ProjectProfile.
  3. Adjust future gates: If a gate consistently fails for a pattern, pre-emptively include that check.
  4. Store as memory entry: Add to Memory Store with effectiveness score for RAG retrieval.
  5. Reviewer feedback integration: Capture human rejection reasons and train agent to pre-empt them.

📝 User Stories

Stories organized by persona: Developer, Team Lead, Security Engineer, and Platform Engineer.

Developer Stories

US-001
As a developer, I want to submit a user story and receive a merge-ready PR, so I can focus on product decisions instead of implementation.
Acceptance Criteria: Given a well-formed user story, when submitted via API or UI, then a PR is created on the target repo within 30 minutes containing passing tests, SAST clean report, and evidence bundle.
US-002
As a developer, I want the agent to understand my existing codebase conventions, so generated code matches the project style.
Acceptance Criteria: Given a cloned repo, when the agent indexes it, then generated code follows detected naming conventions, folder structure, and framework patterns (verified by lint pass).
US-003
As a developer, I want to see the agent's execution plan before it writes code, so I can course-correct early.
Acceptance Criteria: When pre_code approval gate is enabled, the agent pauses after planning and presents the plan in the UI with approve/reject/comment actions.
US-004
As a developer, I want the agent to auto-fix failures and iterate, so I don't have to debug AI-generated code myself.
Acceptance Criteria: When a test or lint check fails, the agent retries up to 5 times with progressive context enrichment; oscillation detection prevents infinite loops.
US-005
As a developer, I want generated tests to cover my acceptance criteria, so I have confidence the PR actually solves the story.
Acceptance Criteria: Each acceptance criterion in the story maps to at least one generated test. Evidence bundle includes the criteria-to-test mapping.
US-006
As a developer, I want to import my existing multi-year repo and start using the agent immediately, not rebuild from scratch.
Acceptance Criteria: Given a git URL, the agent clones, indexes (AST, dependencies, conventions, symbols), and builds a ProjectProfile in under 5 minutes.

Team Lead Stories

US-007
As a team lead, I want to configure which changes need human approval, so high-risk operations are reviewed while routine work flows automatically.
Acceptance Criteria: Policy rules configurable per-repo; auto-merge triggers only when all gates pass AND criticality is below threshold. Infra/auth changes always require review.
US-008
As a team lead, I want a full audit trail of every agent decision, so I can answer "who approved what, when, and why."
Acceptance Criteria: Every decision point logged with actor (agent/human/policy), timestamp, story context, available evidence, and gate evaluated. Exportable as JSON/CSV.
US-009
As a team lead, I want cost budgets per story and per day, so AI spending is predictable and capped.
Acceptance Criteria: Agent pauses on budget threshold; admin receives notification; cost tracking visible in dashboard with per-story and per-day breakdowns.
US-010
As a team lead, I want the agent to learn from reviewer rejections, so the same mistakes don't repeat.
Acceptance Criteria: After human rejection, the agent stores the rejection reason, classifies it, and demonstrates ≥80% reduction in same-class rejections within 10 subsequent stories.
US-011
As a team lead, I want to see a dashboard of agent success rates and iteration counts per repo, so I can measure ROI.
Acceptance Criteria: Dashboard shows: stories processed, merge rate, average iterations, cost per story, quality gate pass rate, all filterable by repo and time range.
US-012
As a team lead, I want the agent to handle coordinated changes across multiple repos in our microservice architecture.
Acceptance Criteria: Given a story that affects shared types/contracts, the agent creates synchronized PRs across affected repos with cross-repo validation before merging.

Security Engineer Stories

US-013
As a security engineer, I want every PR to include a SAST scan report, so vulnerabilities are caught before merge.
Acceptance Criteria: Evidence bundle includes Bandit/Semgrep results; PRs with high/critical findings are blocked; security_finding gate routes to security reviewer.
US-014
As a security engineer, I want secret detection enforced on all generated code, with no override option.
Acceptance Criteria: Secret detection gate is always enforced (not configurable to disable). Regex + entropy scanning catches API keys, tokens, passwords.
US-015
As a security engineer, I want the agent's shell execution sandboxed with command allowlists and env isolation.
Acceptance Criteria: Only pre-approved commands execute; LD_PRELOAD/PYTHONPATH/NODE_OPTIONS blocked; all user input shlex.quote()'d; working directory enforced within project.
US-016
As a security engineer, I want the API protected by authentication with no sensitive data in error responses.
Acceptance Criteria: Bearer token on all API routes; CORS restricted to localhost; error responses return request_id only (no stack traces); 1MB request size limit.
US-017
As a security engineer, I want dependency auditing to catch known CVEs in any packages the agent adds.
Acceptance Criteria: When the agent adds a new dependency, it runs a vulnerability audit; critical CVEs block the PR; severity thresholds configurable.

Platform Engineer Stories

US-018
As a platform engineer, I want to configure LLM providers and model routing without code changes.
Acceptance Criteria: Admin panel allows adding/removing providers, setting API keys, configuring task-to-model mappings, and testing connectivity — all via UI.

Acceptance Criteria: Every log entry includes story_id, request_id, component name, and timestamp; logs parseable by ELK/Datadog/Splunk.

Acceptance Criteria: Dockerfile produces a working container; CPU/memory limits configurable; health endpoint at /health; graceful shutdown on SIGTERM.
US-022
As a platform engineer, I want quality gate templates for common compliance frameworks (SOC2, HIPAA, PCI-DSS).
Acceptance Criteria: Pre-built policy templates loadable via config; each template sets appropriate gate thresholds and required scans for its compliance framework.
US-023
As a platform engineer, I want SSO/SCIM integration for enterprise identity management on approval workflows.
Acceptance Criteria: SAML/OIDC SSO for login; SCIM for user provisioning; approval routing respects organizational roles and team membership.

As a platform engineer, I want post-deploy verification with automatic rollback on failure.

Comprehensive security model covering API authentication, execution sandboxing, generated code safety, and information disclosure prevention.

Research Context Empirical research found meaningful rates of CWE vulnerabilities in AI-generated code across Python and JavaScript. Security scanning is not optional — it must be enforced at every iteration, not just before deployment.

Project Configuration File

  • DEP-2.1: Per-project config file (.autodev.yaml) in repo root specifying: quality gate thresholds, approval gate toggles, cost budgets, LLM routing overrides, and test commands.
  • DEP-2.2: Config inheritance: global defaults → organization defaults → repo-level overrides.
  • DEP-2.3: Quality gate policy templates loadable by name (e.g., policy: soc2-strict).

DEP-3: Container Deployment

  • DEP-3.1: Dockerfile produces a production-ready container with Python 3.11, Node.js, git, and all CLI tools pre-installed.
  • DEP-3.2: Health check at /health endpoint for orchestrator readiness probes.
  • DEP-3.3: Graceful shutdown on SIGTERM — complete in-progress story iteration before exiting.
  • DEP-3.4: Resource limits configurable: CPU, memory, disk, and per-command execution timeouts.
  • DEP-3.5: Docker Compose configuration for single-node deployment with persistent volume for memory store.

DEP-4: Operational Requirements

  • DEP-4.1: Structured logging — All logs in JSON format via structlog, including story_id, request_id, component name, and timestamp. Compatible with ELK/Datadog/Splunk.
  • DEP-4.2: Health monitoring — /health endpoint returns: server uptime, active story count, provider connectivity status, memory store size, queue depth.
  • DEP-4.3: Crash recovery — Story state persisted to disk; on restart, in-progress stories resume from last checkpoint.
  • DEP-4.4: Backup — Memory store and project profiles exportable/importable for disaster recovery.

DEP-5: Rollback & Verification

  • DEP-5.1: Post-deploy verification runs health checks and smoke tests on deployed changes.
  • DEP-5.2: If verification fails, automatic rollback by reverting the PR/deployment with an incident report.
  • DEP-5.3: Verification timeout configurable (default 5 minutes). Timeout treated as failure → rollback triggered.
  • DEP-5.4: Rollback actions logged in audit trail with full evidence of what failed and why.

DEP-6: Scaling Considerations

  • DEP-6.1: Story processing is async — multiple stories can be queued and processed sequentially (parallel processing planned for future).
  • DEP-6.2: LLM calls are the primary bottleneck. Multi-model routing distributes load and enables concurrent calls to different providers.
  • DEP-6.3: Codebase indexing is incremental after initial import — only changed files re-indexed.
  • DEP-6.4: Memory store uses file-backed JSON with Jaccard similarity search. Database-backed store planned for high-volume deployments.
